Shipping an app

So you want to
put an app on phones.

A quick walk through how a small team gets an app onto iPhones and Androids, and what actually happens between "it works on my laptop" and "it's on the App Store." Based on what I learned shipping FIMBY.

By Stephen Rathjen · use ← → to navigate
The single biggest decision

Decide what actually needs to be "native."

There are two very different lanes for shipping changes. The more you can keep out of the native lane, the faster and cheaper your app is to run.

Slow lane

Native shell changes

  • Icons, splash screens, permissions
  • Push notification setup, deep links
  • New native modules or SDKs
  • Anything that touches Apple / Google review

Cost: new build, new review (24 h – 7 days each side), users must update from the store.

Fast lane

Content & UI served at runtime

  • Screens, forms, buttons, colors
  • Business logic, database schema
  • Copy changes, new features
  • Bug fixes to anything above

Cost: deploy + publish. Live in minutes. No store involvement.

FIMBY is an Expo shell around a Salesforce Experience Cloud site. The shell went through the App Store once. Every fix and feature since ships live from Salesforce without touching the stores. That single architectural choice bought us years of shipping velocity.
Step 1 · Turn code into an installable app

Meet EAS Build.

EAS = Expo Application Services. It's Expo's cloud build service. You push a commit; EAS spins up a macOS box for iOS and a Linux box for Android, produces a signed .ipa and .aab, and hands them back.

You never have to own a Mac to build for iOS. You never have to install Android Studio to build for Android. That alone is worth the price of admission for a solo shop.

Think of EAS as the print shop. You bring the manuscript. It gives you back a bound book, correctly formatted for whichever store, ready to hand over.
// eas.json — three build profiles "development": { "developmentClient": true, "distribution": "internal" }, "preview": { "distribution": "internal" }, "production": { "autoIncrement": true }

development — a dev client with hot reload, for you and your team.
preview — a normal build shared privately for testing.
production — the real one; auto-bumps the build number.

The trick is: the same .ipa or .aab can walk up all four rungs. You don't rebuild for each stage — you just move the build between tracks.
Step 3a · The Apple path

iOS: TestFlight, then App Store.

Every iOS release rides the same rails. You upload a build; it processes; you invite testers; you submit for review; Apple decides.

  • Guideline 5.1.1(v) — if users can create accounts, they must be able to delete them inside the app, in ≤ 3 taps.
  • Reviewer test account required — a real login that shows off the app without them signing up.
  • Privacy Nutrition Label — must match your Privacy Policy word for word.
  • User-generated content = 17+ plus documented block / report / mute controls.
  • Encryption disclosure — ITSAppUsesNonExemptEncryption: false if you only use HTTPS.

Step 3b · The Google path

Android: Internal → Closed → Open → Production.

Play has more automated review and fewer human touchpoints. It's usually faster than Apple, but the paperwork is different, not lighter.

  • Data Safety form — Google's version of Apple's privacy label. Same data, reshaped categories.
  • Content rating (IARC) — a questionnaire that spits out global ratings.
  • Target API level — bumps every year. Miss the deadline and updates get blocked.
  • Feature graphic 1024 × 500 — mandatory, and no, a screenshot doesn't count.
  • Account deletion — same rule as Apple now, plus an off-app deletion URL.

Step 4 · The paperwork half

Half the work isn't code.

The build is the easy part. Reviewers spend most of their time on your store listing — the copy, the pictures, the privacy answers. Budget a full week on this alone the first time.

Screenshots & graphics

  • iPhone 6.7" — 1290 × 2796, 8 shots
  • iPad 13" — 2064 × 2752, required if you support tablets
  • Play phone — 1080 × 1920
  • Play tablet — same, re-exported
  • Feature graphic — 1024 × 500, Play only, mandatory
  • App icon — 1024 × 1024 (Apple, no alpha), 512 × 512 (Play)

Listing copy

  • App name (Apple 30 chars)
  • Subtitle (Apple 30) / Short description (Play 80)
  • Promotional text (Apple 170) — editable anytime
  • Keywords (Apple 100, comma-separated)
  • Full description (4 000 chars both stores)
  • Rejection risk: no em-dashes, no false claims, no competitor names

Public URLs & policy

  • Privacy Policy — must return 200 without auth
  • Support URL — real human contact
  • Account deletion page — required by both stores
  • Privacy label (Apple) matches Data Safety form (Play) matches your policy word-for-word
  • Age rating questionnaire — user-generated content = 17+ on iOS
You will spend a full week making screenshots. Real screenshots of the real app. Reviewers reject pure mockups. Build a seeded reviewer account that shows off the app in a warm, populated state — a cold empty screen is what reviewers see by default, and it doesn't sell the product.
Step 5 · Submit day

What actually happens when you press "submit."

Once everything is prepped, submitting is five commands and a few clicks. The tension is all in the review wait.

iOS

  1. Bump version + build number in app.json
  2. eas build --platform ios --profile production
  3. eas submit --platform ios (auto-uploads to App Store Connect)
  4. Attach the build to your app version in App Store Connect
  5. Fill listing, privacy label, reviewer notes
  6. Click Submit for Review. Wait.

Android

  1. Bump versionCode + versionName
  2. eas build --platform android --profile production
  3. eas submit --platform android (uploads to Play Console)
  4. Promote the build from Internal → Production track
  5. Fill Data Safety, content rating, listing
  6. Create release → review runs → phased rollout
The scariest part is waiting for review. The most useful mitigation is a good reviewer guide — a short PDF or HTML page telling the reviewer exactly how to log in, what to try, and where the moderation controls are. My reviewer guide has caught more rejections than any code change.
Step 6 · What "shipping updates" means later

After launch, three update paths — pick the fastest that works.

Instant

Backend / content

  • Database schema
  • Business logic
  • Any screen rendered from the server

Time to live: minutes.
Store involvement: zero.

Fast

JS bundle (Expo Updates OTA)

  • React Native / JS logic fixes
  • UI tweaks in native screens
  • Small feature toggles

Time to live: next app launch.
Store involvement: zero. Users get it silently.

Slow

Native shell changes

  • SDK / Expo version bumps
  • New permissions or entitlements
  • Icon, splash, deep-link scheme

Time to live: 1–7 day review + user updates.
Store involvement: full re-review.

The golden rule: when in doubt, put the change on a faster lane. Every native update is a re-review, a wait, and a chunk of users who never update. The best mobile apps I know look like small shells over big cloud brains.
If I were starting over

Seven things I'd tell past-me.

Read the store rules before writing a line of code

Apple Guidelines 4 (design) & 5 (legal) and Google's Data Safety spec. They'll shape your architecture.

Bake in account deletion from day one

3 taps from home, in-app. Both stores require it. Retrofitting is misery.

Every "clever" native feature is a re-review

Prefer boring. If you can do it in a WebView or on the server, do.

Ship to TestFlight / Internal early

Real devices in real hands catches things simulators never will. Do it before your app is "done."

Screenshots take longer than you think

Budget a full week. Seed a "reviewer account" that shows the app in a warm, populated state.

Write a reviewer guide

A one-pager telling the reviewer how to log in and what to try. Cuts rejections dramatically.

The stores aren't hostile. They're just very literal, and their reviewers see your app cold, in a bland empty account, with no context. Design for that reviewer, and everything else falls into place.
